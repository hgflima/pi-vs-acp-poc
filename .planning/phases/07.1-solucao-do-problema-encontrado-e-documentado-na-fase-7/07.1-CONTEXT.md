# Phase 07.1: Solução do Problema Encontrado e Documentado na Fase 7 - Context

**Gathered:** 2026-04-05
**Status:** Ready for planning

<domain>
## Phase Boundary

Desbloquear SC#3 (OAuth Codex chat produz conteúdo assistente via SSE) e SC#4 (auto-refresh verificado end-to-end via force-expire) da Phase 7, fechando OAUTH-02 e OAUTH-03. O gap é **upstream** do nosso stream-adapter: pi-ai's `openai-codex-responses.js` lança exceção entre linhas 77-168 (antes de empurrar o evento `start`), resultando em `pi-agent-core` emitindo ZERO `message_update` events, resultando em SSE stream que fecha com `event: done` vazio e `errorMessage` silenciosamente perdido. Plan 07-04 já confirmou a Causa B empiricamente e aplicou switch exaustivo defensivo em `stream-adapter.ts` — escopo desta fase é extrair o erro upstream para visibilidade, diagnosticar a causa concreta, aplicar fix adequado (sem gambiarras), e re-verificar SC#3 + SC#4.

</domain>

<decisions>
## Implementation Decisions

### Diagnóstico upstream
- **D-01:** Aplicar o error-surfacing fix em `stream-adapter.ts` (veja D-03) PRIMEIRO e usar a mensagem de erro resultante como ferramenta primária de diagnóstico. Re-rodar UAT SC#3 com OAuth Codex, observar o `event: error` no SSE, extrair `errorMessage`. Se a mensagem revelar causa concreta (ex: "401 unauthorized", "No API key", "model not found"), diagnóstico está completo — parte-se direto para aplicar fix. Se a mensagem for insuficiente (genérica, ambígua, ou ausente), escalar para instrumentação adicional: (a) `curl -v` replicando headers/body exatos do pi-ai contra `chatgpt.com/backend-api/codex/responses`, (b) re-enable `DEBUG_EVENTS=true` + adicionar logging temporário em `node_modules/@mariozechner/pi-ai/dist/providers/openai-codex-responses.js` antes de cada throw point.

### Estratégia de correção
- **D-02:** **Decisão adiada até pós-diagnóstico.** O root cause concreto determina a estratégia apropriada. Princípio orientador: **sem gambiarras** — o fix tem que ser engenharia adequada, não band-aid. Árvore de decisão (a ser aplicada no planejamento após diagnóstico):
  - Root cause em **código nosso** (header errado, scope OAuth faltando, body shape incorreto em payload enviado ao agent) → fix direto em `src/server/`, sem tocar pi-ai.
  - Root cause em **pi-ai, trivial** (header `OpenAI-Beta` faltando, flag de transport errada, `originator` incorreto) → `patch-package` gerando patch comitado em `patches/@mariozechner+pi-ai+0.64.x.patch`. Aplica em `npm install` automaticamente.
  - Root cause em **pi-ai, estrutural** (API endpoint divergente, mudança de shape no ChatGPT backend-api, stream parsing quebrado) → reavaliar: patch-package pesado ainda é preferível a bypass adapter. Bypass adapter em nosso código é **último recurso**, contradiz objetivo de validar pi-ai.

### Surfaçamento de erros no stream-adapter
- **D-03:** Adicionar novo `case "message_end"` ao switch em `src/server/lib/stream-adapter.ts`. Quando `event.message.stopReason === "error"` e `event.message.errorMessage` está presente, emitir SSE event: `event: error`, `data: JSON.stringify({ message: event.message.errorMessage })`. Formato mínimo, consistente com o protocolo SSE atual do projeto (mesmo shape do error emitido no branch `ame.type === "error"` já existente). `stopReason === "aborted"` fica fora deste escopo (usuário cancelou — comportamento atual não emite evento especial, mantém). Fix é permanente (não gated por debug flag) — todo erro futuro de qualquer provider fica visível no cliente sem `done` silencioso.

### Critério de conclusão
- **D-04:** Phase 7.1 fecha quando SC#3 E SC#4 da Phase 7 estão verdes end-to-end, re-verificados via UAT manual com curl (mesmo método de D-08 da Phase 7). Concretamente: (i) `POST /api/chat` com `provider: "openai"` e model Codex (`gpt-5.1` ou similar) produz SSE stream com ≥1 `event: text_delta` contendo conteúdo do assistente; (ii) `POST /api/auth/oauth/debug/force-expire?provider=openai` seguido de `POST /api/chat` completa com sucesso, e `GET /api/auth/status?provider=openai` retorna `oauthExpiry` posterior ao valor forçado — provando que refresh mutex disparou mid-request. OAUTH-02 e OAUTH-03 marcam `[x] Complete` no REQUIREMENTS.md. Phase 7 overall status muda para PASS (07-UAT.md frontmatter e 07-VERIFICATION.md atualizados). Issue/PR upstream no pi-mono **adiada para decisão pós-fix** — quando tivermos patch em mãos, reavaliamos custo/benefício de contribuir de volta.

### Claude's Discretion
- Exato mecanismo de patch-package (versão do package, formato do patch) — a ser decidido ao aplicar.
- Nome e localização de helper function se o `case "message_end"` crescer (atualmente inline é suficiente).
- Se re-habilitar `DEBUG_EVENTS=true` durante investigação; deixar gated off novamente após diagnóstico completo (já é o padrão definido em 07-04).
- Formato da re-verification section em 07-UAT.md e 07-VERIFICATION.md (seguir padrão já estabelecido no Plan 07-04: seção `## Re-verification (Phase 7.1)` apensada ao original).
- Se atualizar comentário `port 53692` leftover em `src/server/routes/oauth.ts:60` (INFO-level anti-pattern capturado em 07-VERIFICATION.md) — oportunístico, não obrigatório.

## Specific Ideas

- **"Sem gambiarras"** — o fix tem que ser engenharia adequada, não band-aid. Bypass adapter (escrever handler Codex próprio substituindo pi-ai) é **último recurso**, contradiz o objetivo do POC de validar pi-ai. `patch-package` é aceito para fixes concretos e revisáveis em pi-ai.
- **Validação é o produto** — o objetivo da Phase 7.1 não é "fazer o chat Codex funcionar" genericamente, é **provar que pi-ai + pi-agent-core sustentam o caso de uso OAuth Codex** (com ou sem patch local). Phase 7 foi quem declarou o gap; Phase 7.1 fecha o gap mantendo a stack original no lugar.
- **Fluxo implícito desenhado durante discussão**:
  1. Implementar D-03 (error surfacing em stream-adapter)
  2. Re-rodar UAT SC#3 → errorMessage aparece no SSE
  3. Diagnosticar causa pela mensagem (ou escalar para curl/instrumentação per D-01)
  4. Aplicar fix apropriado (D-02 resolvido com base no diagnóstico)
  5. Re-rodar SC#3 + SC#4, ambos passam (D-04 cumprido)
  6. (Pós-fix) Reavaliar se vale abrir issue/PR upstream no pi-mono
